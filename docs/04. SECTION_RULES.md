# Section Rules — Dev Portfolio

This document defines section-level rules for the personal developer portfolio.

This document is governed by:

2. STYLING_SCOPE_RULES.md

---

## 1. Purpose

Every section must serve the personal developer portfolio narrative.

This project is NOT a company landing page.

Each section must communicate:

- personal identity
- technical skill
- credibility
- career direction
- project proof
- contact readiness

---

## 2. Final Section Order

The homepage must follow this order:

1. Navbar
2. Hero
3. Who Am I
4. Statistics
5. Career Journey
6. My Core Skill
7. FAQ
8. Why I Stand Out
9. Projects
10. Testimonials
11. Contact
12. Footer

Do not reorder sections without a clear design or narrative reason.

---

## 5. Section Data Rule

Long content must live in src/constants.

Do not hardcode long arrays or repeated content inside section components.

Required constants:

- navigation-data.ts
- stats-data.ts
- career-data.ts
- skills-data.ts
- faq-data.ts
- stand-out-data.ts
- projects-data.ts
- testimonials-data.ts
- social-media-data.ts

Old company-oriented constants must be removed or replaced:

- trusted-companies-data.ts
- services-data.tsx
- services-process-data.ts
- tabs-data.ts

---

## 6. Section Responsibility Rule

Each section component must focus on rendering only.

Allowed inside section components:

- layout composition
- mapping constants to UI
- section-specific visual structure
- minimal local UI state if needed

Not allowed inside section components:

- large hardcoded data
- unrelated business logic
- global styling decisions
- raw hex colors
- uncontrolled inline styles
- reusable primitive logic that belongs in src/components/ui

---

## 7. Styling Scope Rule

Section-specific layout and decoration must stay inside the section component.

Global CSS is only for:

- design tokens
- theme variables
- base reset
- truly reusable utilities

Do not move one-off section styles into globals.css.

---

## 8. Implementation Rule

Work section by section.

Required flow:

1. Audit related files
2. Update constants first
3. Update section component
4. Run build
5. Check git status
6. Commit with professional message

Never edit multiple major sections at once unless explicitly approved.

---

## 9. Quality Standard

Each section must be:

- responsive
- visually consistent
- aligned with dark purple/pink personal branding
- cleanly structured
- data-driven
- free from company copy
- compatible with the design system

Working UI is not enough.
The section must also support long-term maintainability.

---

## 10. Final Principle

A section is valid only if it supports the personal developer story.

If a section looks like a company landing page, it must be rewritten.
