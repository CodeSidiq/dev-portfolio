# STYLING SCOPE RULES

Version: 1.0
Owner: Sidiq Kusumah

Purpose:
This document defines where styling decisions must live.

It exists to prevent scope leakage between:

- global foundation styling
- reusable component styling
- section-specific styling
- decorative implementation details

This document is subordinate to:

- AI_ENGINEERING_CONSTITUTION.md

This document is referenced by:

- PROJECT_RULES_AND_ROADMAP.md
- README_NEXT.md
- section rule documents

If a styling decision violates this document, the decision is invalid.

---

## 0. CORE PRINCIPLE

Global styling must remain foundational.

Section-specific implementation details must remain local.

Reusability must be proven, not assumed.

A class, token, or utility must not be promoted to global scope merely because it "might be reused later".

---

## 1. STYLING LAYERS

All styling decisions must belong to exactly one of these layers.

### A. Global Foundation Layer

Scope:

- design tokens
- semantic theme tokens
- reset/base rules
- typography foundation
- truly reusable layout utilities
- accessibility/focus primitives

Allowed examples:

- `--background`
- `--foreground`
- `--card`
- `--border`
- `--ring`
- `--space-*`
- `--radius-*`
- `--container-max`
- `.ds-container`
- `.ds-focus-ring`

This layer belongs in:

- `src/app/globals.css`

### B. Reusable Component Layer

Scope:

- reusable UI primitives
- reusable component-level variants
- shared interaction styles

Allowed examples:

- Button variants
- card shell primitives
- badge primitives
- reusable layout wrappers
- shared responsive section wrappers

This layer belongs in:

- `src/components/ui/*`
- component-local styling when needed
- Tailwind classes inside reusable components

### C. Section-Local Layer

Scope:

- one-off layout decisions for a specific section
- section-specific decorative positioning
- section-specific media framing
- section-specific text highlight behavior
- section-specific spacing that does not belong to the global scale abstraction

Allowed examples:

- Hero highlight frame behavior
- Hero glow placement
- About-only stats composition
- Skills-only orbit decoration
- Timeline-only node alignment

This layer must live near the section:

- inside section TSX with Tailwind classes
- or in co-located styling for that section

This layer must NOT be promoted to global scope unless reused by multiple sections.

### D. Decorative Layer

Scope:

- purely visual ornaments
- background glow systems
- decorative grids
- ambient overlays
- visual dot patterns

Decorative styling must remain local unless it becomes part of a true shared design primitive.

Decoration must never dominate the global styling layer.

---

## 2. GLOBAL CSS ALLOWLIST

The following are allowed in `src/app/globals.css`:

### Base / Reset

- html/body sizing
- font family assignment
- default background and foreground
- browser-normalization-level rules

### Semantic Theme Tokens

- color tokens
- semantic aliases
- spacing scale
- radius scale
- typography scale tokens
- container tokens

### Truly Reusable Utilities

- container utility
- focus ring utility
- shared rounded helpers
- shared nav height offset only if used across multiple layouts
- shared accessibility helpers

### Theme Mapping

- `@theme`
- semantic mapping for Tailwind-compatible tokens

---

## 3. GLOBAL CSS DENYLIST

The following must NOT live in `src/app/globals.css` unless explicitly approved by constitution update:

### Section-Specific Tokens

Examples:

- `--hero-*`
- `--about-*`
- `--skills-*`
- `--timeline-*`
- `--faq-*`
- `--projects-*`
- `--contact-*`

### Section-Specific Utility Classes

Examples:

- `.ds-hero-*`
- `.ds-about-*`
- `.ds-skills-*`
- `.ds-timeline-*`

### Decorative Positioning Rules for One Section

Examples:

- one-off glow coordinates
- one-off dot coordinates
- one-off image frame transforms
- one-off highlight offsets

### Micro-layout Text Helpers Bound to One Section

Examples:

- heading prefix offsets
- ampersand-specific alignment helpers
- highlight stack adjustments for a single hero headline

If a styling rule is meaningful only inside one section, it is not global.

---

## 4. TOKEN SCOPE RULES

Tokens must be classified before being created.

### A. Foundation Tokens

Reusable across the whole project.

Examples:

- colors
- spacing scale
- radius scale
- typography scale
- container width
- standard z-index layers if globally reused

These may live in `:root`.

### B. Component Tokens

Reusable across more than one component but not necessarily the whole page.

Examples:

- shared CTA dimensions
- shared icon button sizing
- shared badge sizing

These should be placed carefully.
Default preference:

- keep them near reusable components
- only promote to root when cross-section reuse is proven

### C. Section Tokens

Used by exactly one section.

Examples:

- hero media width
- hero-specific copy max width
- timeline node offset
- about visual frame spacing

These must NOT live in `:root`.

They must stay local to the section implementation.

---

## 5. REUSABILITY TEST

Before promoting any class, token, or utility to global scope, all of the following must be answered:

1. Is it used in at least two separate sections or reusable components?
2. Does it represent a real system rule, not a one-off visual adjustment?
3. Would moving it global make the system clearer, not more confusing?
4. Would another engineer reasonably discover and reuse it correctly?
5. Does it reduce duplication without introducing abstraction noise?

If the answer is "no" to any of the above, keep it local.

---

## 6. TAILWIND POLICY

### Default Rule

Use Tailwind utilities directly in components and sections when they are readable and local.

### Token Rule

Use semantic tokens for:

- colors
- radius
- typography references
- shared spacing scale

### Arbitrary Value Rule

Arbitrary Tailwind values are:

- forbidden by default
- allowed only when justified by design fidelity
- allowed only when the value is not cleanly representable by existing tokens/utilities
- allowed only when usage remains readable and localized

Examples of acceptable controlled usage:

- one-off `max-w-[68.75rem]` for Figma fidelity
- one-off `tracking-[-0.02em]` for heading accuracy
- one-off `top-[42px]` for decorative alignment

Examples of unacceptable usage:

- arbitrary values scattered everywhere without system reasoning
- arbitrary colors instead of tokens
- arbitrary spacing replacing the spacing scale carelessly

---

## 7. INLINE STYLE POLICY

Inline styles are forbidden by default.

Allowed only when all conditions are met:

- value is truly dynamic at runtime
- Tailwind/class-based styling is not appropriate
- CSS variables are not a cleaner alternative
- usage is documented and justified

Accepted cases:

- runtime transform values
- runtime measured positioning
- CSS variable injection for dynamic animation/math

Rejected cases:

- static color values
- static spacing
- static width/height
- static gradients that belong in tokens/utilities

---

## 8. SECTION IMPLEMENTATION RULE

Each section must own its one-off styling.

That means:

- Hero owns Hero-only styling
- About owns About-only styling
- Skills owns Skills-only styling
- Timeline owns Timeline-only styling

Section-local styling may exist:

- in TSX via Tailwind
- in a co-located CSS/module file
- via local constants if needed

Section-local styling must not silently expand the global layer.

---

## 9. DECORATIVE ELEMENT RULE

Decorative elements must be classified before implementation:

- global decorative primitive
- section-local decoration

Default assumption:

- decorations are section-local

Global promotion is allowed only if the same decorative pattern is intentionally reused across the project.

---

## 10. RESPONSIVE RULE

Responsive behavior must respect scope as well.

Global responsive tokens may define:

- shared typography scale
- shared container behavior
- shared nav height behavior

Section-local responsive logic must remain inside the section when it affects:

- composition
- overlay alignment
- hero media framing
- local decorative scaling

Responsive rules must not be centralized unnecessarily.

---

## 11. ENFORCEMENT EXAMPLES

### Valid Global

- `--background`
- `--foreground`
- `--space-4xl`
- `.ds-container`
- `.ds-focus-ring`

### Invalid Global

- `--hero-copy-max`
- `.ds-hero-heading-row-second`
- `.ds-hero-dot-desktop-4`
- `.ds-hero-glow-top-left`

### Valid Local

- Hero-specific glow placement in `Hero.tsx` or local Hero styling
- Timeline node offsets in timeline-local styling
- Skills orbit positioning near the Skills section

---

## 12. FINAL LAW

Global scope is not a convenience dump.

Global scope is foundational infrastructure.

Anything that is:

- one-off
- section-bound
- decorative-detail-heavy
- micro-layout-specific

must remain local until proven reusable.

A styling system that centralizes everything becomes harder to reason about, not easier.
