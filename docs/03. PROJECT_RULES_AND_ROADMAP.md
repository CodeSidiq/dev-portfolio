## 0. PROJECT GOAL

Build a production-ready, dark-themed, responsive portfolio landing page with:

- Next.js App Router
- TypeScript
- Tailwind CSS v4
- shadcn/ui
- Framer Motion
- next/image
- Zod where form validation is required

Primary goals:

- pixel-faithful design implementation
- scalable folder structure
- clear engineering discipline
- predictable styling scope
- data-driven content
- maintainable UI architecture

This project is not a throwaway demo.

Rules:

- `components/layout/` contains persistent layout structures
- `components/sections/` contains page sections
- `components/ui/` contains reusable UI primitives
- `constants/` contains static content and structured data
- `lib/` contains utilities/helpers only

## 5. UI ELEMENT CLASSIFICATION (MANDATORY)

Before implementing any UI element, it must be classified into exactly one of these categories.

### 5.1 Fixed Layout Elements

Elements that must maintain strict visual stability and follow Figma positioning closely.

Examples:

- navbar
- hero structural composition
- section containers
- media frames
- headline highlight frames

Typical implementation tools:

- rem
- px
- design tokens
- controlled local utilities

These elements must NOT scale freely with viewport.

### 5.2 Responsive Layout Elements

Elements that adapt to viewport size while preserving proportions.

Examples:

- container width
- section vertical spacing
- responsive typography
- background pattern regions

Preferred techniques:

- clamp()
- min()
- max()
- vw/vh when justified

Responsive scaling must remain proportional to the design reference.

### 5.3 Decorative Elements

Purely visual elements that do not define layout structure.

Examples:

- glow effects
- grid patterns
- background ornaments
- gradient overlays
- ambient shapes

Decorative elements may scale responsively, but must NEVER break layout composition.

Default rule:
Decorative elements are section-local unless proven reusable.

### 5.4 Content Elements

Elements driven by text or data.

Examples:

- headings
- paragraphs
- buttons
- labels
- metadata
- cards driven by constants

Content elements must preserve readability across breakpoints.

### 5.5 Classification Rule

UI classification must happen before implementation begins.

Classification determines:

- styling scope
- responsive behavior
- reusability expectation
- placement in global vs local styling

## 6. STYLING SCOPE RULE

This project follows strict scope-based styling governance.

### Global layer may contain only:

- design foundation tokens
- semantic theme tokens
- base/reset
- truly reusable utilities
- shared accessibility primitives

### Local layer must contain:

- section-only layout details
- section-only decorative rules
- section-only token values
- section-only positional adjustments
- section-only highlight/glow/ornament behavior

Examples:

- `--background` may be global
- `.ds-container` may be global
- `--hero-copy-max` must be local
- `.ds-hero-glow-top-left` must be local

See:

- STYLING_SCOPE_RULES.md

---

## 7. STYLING POLICY

### Required

- no raw hex in TSX
- use semantic tokens for color
- spacing must follow scale by default
- radius must follow scale by default
- typography must respect design system
- local section styling must not leak into global scope

### Controlled Allowances

Arbitrary Tailwind values are allowed only when:

- design fidelity requires them
- token/system alternatives are not appropriate
- usage is local, sparse, and readable

Inline styles are allowed only when:

- values are runtime-dynamic
- class/token approaches are not suitable
- usage is explicitly justified

See:

- AI_ENGINEERING_CONSTITUTION.md
- STYLING_SCOPE_RULES.md
- DESIGN_SYSTEM.md

## 8. CLIENT / SERVER DISCIPLINE

Default:

- Server Components first

Use `"use client"` only when required for:

- state
- event handling
- animation
- browser APIs

Avoid unnecessary client boundaries.

---

## 9. ACCESSIBILITY AND MOTION

### Accessibility

- semantic HTML required
- keyboard access required
- icon-only buttons need labels
- Escape closes overlays where applicable
- body scroll lock required for fullscreen mobile overlays
- focus handling required where overlays/dialogs exist

### Motion

- Framer Motion is primary
- CSS transitions allowed for micro-interactions
- motion must remain subtle and purposeful
- motion must not break layout stability

---

## 10. ASSET POLICY

All assets must live inside:

- `public/assets/`

Disallowed:

- remote placeholder services
- duplicated inline SVGs without reason
- random external image URLs
- asset scattering outside defined folders

---

## 11. CONSTITUTIONAL GUARDRAILS

The following patterns are invalid:

- global CSS becoming a section styling dump
- section-specific tokens promoted to `:root` without proven reuse
- global classes created for one-off decorative details
- content duplicated outside constants
- arbitrary styling without controlled justification

A working result that violates scope discipline must be rejected.
